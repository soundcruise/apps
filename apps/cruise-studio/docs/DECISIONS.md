## ADR-001: apps/cruise-studio/ の1アプリ内モジュール構成にする

- 日付: 2026-07-02 / 状態: 採用

**背景**: 譜面クルーズとプレDTMクルーズを `apps/sheet-cruise/` のような
別アプリフォルダに分ける案もあった。

**決定**: アプリは `apps/cruise-studio/` の1つ。譜面クルーズ / プレDTMクルーズは
`js/modules/` 配下の内部モジュールとして持つ。

**理由**:
- 両モジュールは同じ StudioProject・同じ localStorage・同じ編集セッションを扱う。
  別アプリに分けるとデータ共有が「エクスポート/インポート」になり構想と矛盾する
- 別フォルダ案は service-worker / manifest / バージョン管理が倍増するだけで利点がない
- 既存Sound Cruiseアプリ群も「1アプリ=1フォルダ、内部に機能同居」が基本形

## ADR-002: 当面 service-worker / manifest を作らない

- 日付: 2026-07-02 / 状態: 採用

**決定**: MVP期間中は service-worker も manifest.json も作らない。PWA化しない。

**理由**:
- クルーズスタジオはPC用ブラウザアプリであり、ホーム画面追加・オフライン動作の必要が薄い
- Sound Cruise シリーズでは過去にキャッシュ更新まわりが事故源になり得るため、
  リスク源を最初から持たないのが最善
- 将来必要になれば rhythm-cruise の network-first 最小SW（キャッシュ非保持・
  自プレフィックスのみ掃除・スコープ `./` 限定）をそのまま移植できる
- 登録する場合もスコープは必ず `apps/cruise-studio/` 内。ルートスコープ登録は永久に禁止

## ADR-003: 初期MVPにPRO認証を入れない

- 日付: 2026-07-02 / 状態: 採用

**決定**: PRO認証はPhase 8まで実装しない。通常版/PRO版のディレクトリも今は分けない。

**理由**:
- 譜面クルーズは課金価値を検証する前の段階
- リズムクルーズの前例（通常版＝アプリルート直下で開発し、PRO版は後から
  `pro_xxx/` サブフォルダを追加して `../script.js` 等を共用）が実証済みで、
  後から分けるコストがほぼゼロ
- 導入時は `shared/pro-gate.js` + インライン `__SOUNDCRUISE_PRO_GATE__` 設定に
  そのまま乗る（sharedは変更しない）。localStorage `soundCruiseProAuth` により
  既存3アプリと解錠状態が共通になる。これが既存思想への最も安全な合流方法
- プラン情報・featureFlags は曲データ（StudioProject）には入れず、
  `cruiseStudio.appSettings` 側に持つ

## ADR-004: 位置・音価は tick 方式（ticksPerBeat=480）で持つ

- 日付: 2026-07-02 / 状態: 採用

**決定**: コード・歌詞・メロディ・ストロークの位置と長さを、すべて
「小節番号＋小節内tick」で持つ。ticksPerBeat は標準MIDI解像度の480。
編集UIは16分グリッド（120tick）量子化を基本とする。

**理由**:
- 「譜面は曲データのビュー、MIDIは曲データの変換」という中心思想の土台。
  見た目座標を持つと譜面専用データ化し、プレDTM生成に使えなくなる
- 歌詞とドレミが同じtick座標系に乗るため、位置合わせ問題（このアプリ最大の
  技術リスク）がデータ構造レベルで解決する
- 480はSMF書き出し時に変換不要。3連符（160tick）も整数で表せる
- 自由配置（px指定）を許すと編集・印刷・MIDIの三者で座標系が分裂する

## ADR-005: PCファーストUIにする

- 日付: 2026-07-02 / 状態: 採用

**決定**: クルーズスタジオはPC（広い画面＋キーボード）前提で設計する。
スマホは「表示が完全崩壊しない」最低限のみ。既存スマホアプリのPC化改修もしない。

**理由**:
- 中心作業が譜面作成・歌詞入力・ドレミ入力・小節グリッド編集・PDF出力・
  MIDI書き出し・将来のDAW連携であり、いずれも広い画面とキーボードが前提
- 既存シリーズの `shared/style.css` は max-width 600px のスマホ前提で、
  この用途と相容れない。rhythm-cruise と同様に shared CSS には依存せず
  theme.css 1枚で構成する（デザイン思想はDESIGN_SYSTEM.mdから継承する）

## ADR-006: マリーゴールドPDFはレイアウト参考のみ。実データには使わない

- 日付: 2026-07-02 / 状態: 採用

**決定**: 参考PDF（マリーゴールド/あいみょん）は「教材コード譜の紙面構造」の
参照にのみ使う。歌詞・メロディ・ドレミなどの中身を、サンプルデータ・テストデータ・
実装データとして repo に入れない。

**理由**:
- 歌詞・メロディは著作物であり、公開リポジトリ／公開サイトに含めると権利侵害になる
- レイアウト構造（ヘッダー情報・コードグリッド・コード＋歌詞＋ドレミの3段組）は
  アイデアであり、参照して独自実装する分には問題ない
- サンプルが必要な場合は架空の短い曲（`createSampleProject()`）を使う

## ADR-007: JSはビルドなしのクラシック script 複数本＋グローバル名前空間

- 日付: 2026-07-02 / 状態: 採用

**決定**: ES modules もバンドラも使わず、`<script src="...?v=x.y.z">` を
読み込み順に並べる。各ファイルは `window.CruiseStudio` 名前空間へ登録する。

**理由**:
- リポジトリ全体が「ビルドなし静的ホスティング＋ `?v=` キャッシュバスティング」の
  思想で運用されており、ES modules は import 先URLの `?v=` 管理と相性が悪い
- rhythm-cruise の script.js 1本31,000行超の教訓から、最初から
  core / modules 単位でファイル分割だけは行う
- 読み込み順は index.html が正: music-theory → song-model → storage → app
