## ADR-001: apps/cruise-studio/ の1アプリ内モジュール構成にする

- 日付: 2026-07-02 / 状態: 採用

**背景**: 譜面クルーズとプレDTMクルーズを `apps/sheet-cruise/` のような
別アプリフォルダに分ける案もあった。

**決定**: アプリは `apps/cruise-studio/` の1つ。譜面クルーズ / プレDTMクルーズは
`js/modules/` 配下の内部モジュールとして持つ。

**理由**:
- 両モジュールは同じ StudioProject・同じ localStorage・同じ編集セッションを扱う。
  別アプリに分けるとデータ共有が「エクスポート/インポート」になり構想と矛盾する
- 別フォルダ案は service-worker / manifest / バージョン管理が倍増するだけで利点がない
- 既存Sound Cruiseアプリ群も「1アプリ=1フォルダ、内部に機能同居」が基本形

## ADR-002: 当面 service-worker / manifest を作らない

- 日付: 2026-07-02 / 状態: 採用

**決定**: MVP期間中は service-worker も manifest.json も作らない。PWA化しない。

**理由**:
- クルーズスタジオはPC用ブラウザアプリであり、ホーム画面追加・オフライン動作の必要が薄い
- Sound Cruise シリーズでは過去にキャッシュ更新まわりが事故源になり得るため、
  リスク源を最初から持たないのが最善
- 将来必要になれば rhythm-cruise の network-first 最小SW（キャッシュ非保持・
  自プレフィックスのみ掃除・スコープ `./` 限定）をそのまま移植できる
- 登録する場合もスコープは必ず `apps/cruise-studio/` 内。ルートスコープ登録は永久に禁止

## ADR-003: 初期MVPにPRO認証を入れない

- 日付: 2026-07-02 / 状態: 採用

**決定**: PRO認証はPhase 8まで実装しない。通常版/PRO版のディレクトリも今は分けない。

**理由**:
- 譜面クルーズは課金価値を検証する前の段階
- リズムクルーズの前例（通常版＝アプリルート直下で開発し、PRO版は後から
  `pro_xxx/` サブフォルダを追加して `../script.js` 等を共用）が実証済みで、
  後から分けるコストがほぼゼロ
- 導入時は `shared/pro-gate.js` + インライン `__SOUNDCRUISE_PRO_GATE__` 設定に
  そのまま乗る（sharedは変更しない）。localStorage `soundCruiseProAuth` により
  既存3アプリと解錠状態が共通になる。これが既存思想への最も安全な合流方法
- プラン情報・featureFlags は曲データ（StudioProject）には入れず、
  `cruiseStudio.appSettings` 側に持つ

## ADR-004: 位置・音価は tick 方式（ticksPerBeat=480）で持つ

- 日付: 2026-07-02 / 状態: 採用

**決定**: コード・歌詞・メロディ・ストロークの位置と長さを、すべて
「小節番号＋小節内tick」で持つ。ticksPerBeat は標準MIDI解像度の480。
編集UIは16分グリッド（120tick）量子化を基本とする。

**理由**:
- 「譜面は曲データのビュー、MIDIは曲データの変換」という中心思想の土台。
  見た目座標を持つと譜面専用データ化し、プレDTM生成に使えなくなる
- 歌詞とドレミが同じtick座標系に乗るため、位置合わせ問題（このアプリ最大の
  技術リスク）がデータ構造レベルで解決する
- 480はSMF書き出し時に変換不要。3連符（160tick）も整数で表せる
- 自由配置（px指定）を許すと編集・印刷・MIDIの三者で座標系が分裂する

## ADR-005: PCファーストUIにする

- 日付: 2026-07-02 / 状態: 採用

**決定**: クルーズスタジオはPC（広い画面＋キーボード）前提で設計する。
スマホは「表示が完全崩壊しない」最低限のみ。既存スマホアプリのPC化改修もしない。

**理由**:
- 中心作業が譜面作成・歌詞入力・ドレミ入力・小節グリッド編集・PDF出力・
  MIDI書き出し・将来のDAW連携であり、いずれも広い画面とキーボードが前提
- 既存シリーズの `shared/style.css` は max-width 600px のスマホ前提で、
  この用途と相容れない。rhythm-cruise と同様に shared CSS には依存せず
  theme.css 1枚で構成する（デザイン思想はDESIGN_SYSTEM.mdから継承する）

## ADR-006: マリーゴールドPDFはレイアウト参考のみ。実データには使わない

- 日付: 2026-07-02 / 状態: 採用

**決定**: 参考PDF（マリーゴールド/あいみょん）は「教材コード譜の紙面構造」の
参照にのみ使う。歌詞・メロディ・ドレミなどの中身を、サンプルデータ・テストデータ・
実装データとして repo に入れない。

**理由**:
- 歌詞・メロディは著作物であり、公開リポジトリ／公開サイトに含めると権利侵害になる
- レイアウト構造（ヘッダー情報・コードグリッド・コード＋歌詞＋ドレミの3段組）は
  アイデアであり、参照して独自実装する分には問題ない
- サンプルが必要な場合は架空の短い曲（`createSampleProject()`）を使う

## ADR-007: JSはビルドなしのクラシック script 複数本＋グローバル名前空間

- 日付: 2026-07-02 / 状態: 採用

**決定**: ES modules もバンドラも使わず、`<script src="...?v=x.y.z">` を
読み込み順に並べる。各ファイルは `window.CruiseStudio` 名前空間へ登録する。

**理由**:
- リポジトリ全体が「ビルドなし静的ホスティング＋ `?v=` キャッシュバスティング」の
  思想で運用されており、ES modules は import 先URLの `?v=` 管理と相性が悪い
- rhythm-cruise の script.js 1本31,000行超の教訓から、最初から
  core / modules 単位でファイル分割だけは行う
- 読み込み順は index.html が正: music-theory → song-model → storage → app

## ADR-008: Phase 2Bの編集UIは「1小節1コード・警告のみ検証・4/4前提」で開始する

- 日付: 2026-07-02 / 状態: 採用

**決定**: 譜面クルーズの最初の小節グリッド編集は以下の制限付きで開始する。

1. **1小節1コード**: コード入力は小節ごとに1つ（`chords[0] = {tick:0, durationTicks:小節長}`）。
   データ構造（`chords[]` 配列＋tick）は最初から複数コード対応なので、
   Phase 2C以降でUIだけを拡張すればよい。モデル層の `setBarChord()` にこの制限を閉じ込める
2. **コード検証は警告のみ**: `parseBasicChordSymbol()` で解釈できない表記は
   警告表示（アンバー枠）にするが、入力も保存も禁止しない。
   実際のコード表記は多様であり、初期MVPで厳密に拒否するとユーザーの入力を阻害する
3. **4/4前提で開始**: 拍子UIは4/4のみ。小節長は `getBarLengthTicks()` に集約してあり、
   将来の拍子対応はこの関数と拍子selectの拡張で行う

**理由**: 教材コード譜の大半は「1小節1〜2コード・4/4」で書けるため、
最小の編集UIで実用に到達させ、複雑さはデータ構造ではなくUI側の段階拡張で吸収する。

## ADR-009: 歌詞・ドレミのMVP入力は既存イベント形式へ正規化して保存する

- 日付: 2026-07-03 / 状態: 採用

**決定**:

1. **歌詞（案A採用）**: 1小節分の歌詞テキストを `lyrics[]` の **tick:0 単一イベント**
   （`[{tick: 0, text: "1小節分の文字列"}]`）として保存する。
   `lyricsText` のような補助フィールド（案B）は追加しない。
2. **ドレミ**: 入力文字列は保存せず、簡易パーサで **melody イベント
   （tick / durationTicks / step / alter / octave）へ正規化**して保存する。
   - 音名: ド〜シ（ひらがな可）、♯/♭、オクターブ ↑↓、伸ばし 〜/ー/-
   - 配置: 音数が8分で収まれば8分スロット、超えれば16分スロットに先頭から詰める
   - 伸ばし記号は直前の音の durationTicks を1スロット延長する
   - 解釈できない文字は警告表示して破棄する（保存はできる）

**理由**:
- 案Bや文字列フィールド追加は schemaVersion 変更＋migration＋二重の正を生む。
  案A/イベント正規化なら**スキーマ不変・migration不要**で、将来の tick 分割入力は
  「1イベント→複数イベント」への granularity 変更だけで済む（イベントの形は同じ）
- melody は将来のMIDI生成・五線譜・TAB・MusicXMLの本命データであり、
  文字列で持つと後で全データの再パースが必要になる。今から数値イベントで持つ方が安全
- トレードオフ: 入力欄の値はイベントから再生成されるため、未知文字や余分な空白は
  再表示時に正規化される（許容。UI注記あり）

## ADR-010: 印刷は print.css 分離のブラウザ印刷で始める

- 日付: 2026-07-03 / 状態: 採用

**決定**: PDF出力はブラウザ印刷（`window.print()`）ベースとし、
印刷スタイルは `theme.css` 内の `@media print` ではなく **`print.css` に分離**して
`media="print"` で読み込む。印刷対象は譜面プレビュー（`#sc-preview`）のみで、
編集UI・ナビ・ツールバーは `display: none` で除外。`@page` でA4縦・余白を指定する。

**理由**:
- 画面用 `theme.css` は今後も肥大するため、紙面規則を独立ファイルに隔離した方が
  Phase 3 の「A4固定px紙面（794×1123px）」への移行時に差し替えやすい
- 専用PDFライブラリはMVPでは過剰。ブラウザ印刷で教材配布には十分
- 既知の制約: 印刷は譜面クルーズ画面を開いている時のみ有効（TOP画面での⌘Pは空になる）

## ADR-011: アプリ全体は黒基調の高級感、紙面だけは白

- 日付: 2026-07-03 / 状態: 採用

**決定**: クルーズスタジオのUIは「黒〜焦げ茶の深い背景＋アンバー/ゴールドの
アクセント＋暗色グラデーションのカード＋細い枠線＋控えめなシャドウ」で統一する
（リズムクルーズ DESIGN_SYSTEM.md の Clean Pro 系を継承）。
ただし**譜面プレビュー（紙面）だけは白系**とし、印刷物としての可読性を優先する。
「暗いスタジオの中に白い譜面が置いてある」構図をこのアプリの基本画面構成とする。

**理由**: 制作スタジオらしい落ち着きと、教材コード譜としての実用性を両立するため。
フォーカスリングはアンバーの淡いグローで統一し、入力フォームも黒背景に馴染ませる。

## ADR-012: 既存アプリは本体無変更のまま「別タブリンク起動」を正式導線にする

- 日付: 2026-07-03 / 状態: 採用

**決定**: 音感 / 指板 / リズムクルーズへの導線は、クルーズスタジオ内のランチャー画面
（アプリ名・用途説明・注意書き・「新しいタブで開く」）からの**別タブリンク起動**とする。
リンクは同一オリジンの相対パス（`../pitch-cruise/standard/` 等）。
**iframe埋め込みは採用しない**。

**理由**:
- 既存アプリ本体を一切変更しない、が絶対条件。リンク起動は既存側の変更ゼロで成立する
- iframeはマイク権限（音感・リズム）、service workerのスコープ、スマホ縦画面前提の
  viewport設定、PROゲートのオーバーレイ表示と干渉するリスクが複数あり、
  検証コストに見合わない
- PC向けの見せ方はクルーズスタジオ側のカード・説明文で整え、既存アプリが
  スマホ向けであることを明示して期待値を揃える
- 将来の連携はURLパラメータ受け渡し（既存側の対応が必要になった時点で別途合意）から始める

## ADR-013: プレDTMクルーズMVPは「音を鳴らす」より「生成イベントの確認」を優先する

- 日付: 2026-07-03 / 状態: 採用

**決定**: プレDTMクルーズの最初の実装は、StudioProject から
アコギ / ベース / ドラムの伴奏イベントを生成し、**小節ごとの表で確認できる**ところまでとする。
Web Audio再生・SMF(.mid)バイナリ書き出しはこのフェーズでは実装しない。
生成結果はJSONファイルとして書き出せる。

ただし生成イベントは最初から **絶対tick・durationTicks・MIDIノート番号・ベロシティ** を持ち、
SMF書き出し・試聴はこのイベント列の変換だけで実装できる形にしておく（実質的なMIDI土台）。

生成規則（MVP）:
- アコギ = コード × ストロークパターン（down/up/mute、アクセントでベロシティ差）
- ベース = ルート音（オンコードはベース音）を8分刻み、1・3拍を強く。実音は capo を加味
- ドラム = 基本8ビート（kick 1/3拍・snare 2/4拍・hihat 8分。GMマップ 36/38/42）
- コード未入力の小節は直前コードを継続（リードシート慣習）。曲頭の無コード区間は休み

**理由**:
- 母艦の大枠フェーズでは「同じ曲データが伴奏になる」ことの確認が目的であり、
  音質・再生実装に踏み込むと検証範囲が膨らむ
- SMFバイナリの正しさはDAWでの実機確認が必要で、このフェーズの検証手段を超える。
  イベント列さえ正しければ、SMF化は純粋な変換問題として次フェーズで安全に足せる
- 生成を純関数（プロジェクト無変更）に保つことで、譜面クルーズ側への影響をゼロにする

## ADR-014: 伴奏設定は曲データではなくアプリ設定（cruiseStudio.appSettings）に保存する

- 日付: 2026-07-03 / 状態: 採用

**決定**: プレDTMクルーズの伴奏設定（パートON/OFF・パターン・アクセント）は、
StudioProject の `arrangement` フィールドには保存せず、
localStorage の `cruiseStudio.appSettings` 内 `preDtmSettings` に保存する。
`generateArrangement(project, settings)` は settings を引数で受け、曲データを変更しない。

**理由**:
- StudioProject に書き込むと、譜面クルーズの未保存管理（dirty）とスキーマ管理
  （フィールド追加＝schemaVersion更新規約）に波及し、リスクに見合わない
- MVPでは「設定を変えて生成結果を確認する」ことが目的で、設定は全プロジェクト共通で十分
- 将来プロジェクト固有のアレンジ保存が必要になったら、`arrangement` フィールドへの
  移行（schemaVersion 2 + migrateProject）として別途行う。この移行はアプリ設定→
  曲データへのコピーだけで済む

## ADR-015: MIDI書き出しはSMF Type 1自前実装。対象はベース/ドラム先行、アコギは除外

- 日付: 2026-07-03 / 状態: 採用

**決定**: `.mid` 書き出しを実装する。外部ライブラリは追加せず、
`midi-export.js` にSMF Type 1の最小エンコーダを自前実装する。

- トラック構成: Track 0=メタ（曲名/テンポ/拍子）、Track 1=ベース（ch1・
  Electric Bass (finger)）、Track 2=ドラム（ch10・GMマップ 36/38/42）
- **アコギは今回MIDI化しない**。書き出し対象を画面に明記し、
  ストロークイベント（コード記号・方向・アクセント・ベロシティ）は伴奏JSONに全量含める
- 生成バイト列は `inspectSmf()`（チャンク構造・ヘッダ・ノート数の簡易パーサ）で
  ダウンロード前に自己検証する。DAWでの実機確認はユーザー側で行う

**理由**:
- ベースとドラムはMIDIノート番号・ベロシティ・絶対tickが確定しており、
  SMF化は純粋なバイナリ変換で安全に実装できる（依存追加も不要な規模）
- アコギは「コード記号→実音ノート列」のボイシング展開が未実装であり、
  ルート単音などで無理にノート化すると質の低いMIDIになる。
  対象外であることを明示する方が、DAWに持ち込むユーザーにとって誠実
- ボイシング展開（将来フェーズ）が入れば、アコギも同じイベント形式で
  Track 3 として追加するだけでよい

## ADR-016: A4紙面は「画面px＋スケールフィット / 印刷mm」のハイブリッドで固定する

- 日付: 2026-07-04 / 状態: 採用

**決定**: 譜面クルーズのプレビューをA4固定紙面レンダラー（`sheet-renderer.js`）に置き換える。

1. **サイズ方式**: 画面では `.sheet-page` を **794×1123px（A4@96dpi）** で組み、
   プレビューペインの幅に `transform: scale()` でフィットさせる（横スクロールさせない。
   ビューポートの高さはJSが scale後の実寸で確保する）。
   印刷では print.css が **210mm × min-height 297mm・padding 14mm/15mm** のmm指定へ寄せる。
   96dpi換算で両者は同比率のため、画面プレビューと印刷結果のズレが最小になる。
   `@page` は `margin: 0`（余白は紙面のpaddingが持つ。marginを持たせると width:210mm がはみ出すため）
2. **DOM構造**: `.sheet-page`（ページ）> ヘッダー / セクション見出し / `.sheet-line`（譜面行）>
   `.sheet-bar`（小節セル）。譜面行を1要素にすることで `break-inside: avoid` が行単位で効き、
   印刷時に行の途中で改ページしない。複数ページ対応は「render()がページ配列を返す」拡張で行う
   （現状は1ページに流し、あふれた分は印刷時にCSSの改ページ制御で次ページへ流れる）
3. **barsPerLine / lineBreakAfter（案A採用）**: 1行の小節数は `sheetSettings.barsPerLine` を正式採用
   （歌詞・ドレミが出ないセクションは2倍詰め）。`bars[].lineBreakAfter` はスキーマに保持するが
   **現状のレンダラーでは未使用**。手動改行UIを導入するフェーズで「barsPerLine=既定、
   lineBreakAfter=手動上書き」として使う。UIがない今フラグを効かせると、
   小節数変更後に古いフラグが残ってレイアウトが崩れる事故の方が怖い
4. **表示トグル**: `sheetSettings.showChords / showLyrics / showDoremi` をチェックボックスで
   切り替え可能にした（曲データの一部として保存される）。`showStrum` は紙面のストローク記号
   表示が未実装のためトグル対象外（実装時に追加）

**理由**: 「画面＝印刷」の原則を最小コストで成立させ、教材配布（PDF）に耐える紙面を
既存データ（tickイベント）から純粋にレンダリングするため。

## ADR-017: 紙面タイミンググリッドは紙面DOMの一部として描き、設定は補完で読む（S1a）

- 日付: 2026-07-04 / 状態: 採用

**決定**: 譜面の小節セル内に8分/16分のタイミンググリッド線を表示する（v0.8.0）。

1. **描画方式**: グリッド線は `sheet-renderer.js` が `.sheet-bar` の背面
   （`.sheet-bar-grid`、z-index:-1）に紙面DOMの一部として生成する。
   ONなら画面にも印刷/PDFにも出る。OFFなら要素自体を生成しない
   （print.css での隠し分岐を持たない。「画面＝印刷」の原則を守る）
2. **濃淡**: 小節線（セル枠）＞ 拍線 ＞ 8分線 ＞ 16分線。
   印刷用の線色は print.css 側で独立して微調整できるようにしておく
3. **設定**: `sheetSettings.showTimingGrid`（既定 true）と
   `sheetSettings.timingGridResolution`（8 | 16。既定 8）。
   表示専用フィールドのため **schemaVersion は 1 のまま**とし、
   欠損時は `resolveShowSettings()` が既定値で補完する（DATA_MODEL.md 8章の例外規約）
4. **小節単位の16分自動判定**: 全体設定が8分でも、16分位置（120tick粒度）の
   lyrics / chords / melody イベントを含む小節は自動的に16分グリッドで表示する。
   分割状態はデータに保存せず、常にtickから導出する
5. **将来の編集レイヤー**: 画面専用の編集オーバーレイ（S1b以降の `.slot-overlay`）は
   print.css で `display: none !important` により印刷から除外する枠を先に用意した

**理由**: 次フェーズの譜面上直接編集（スロット入力）で8分/16分の位置が見えることが
前提になるため、まずグリッド線だけを安全に導入する。紙面DOMの一部として描くことで
画面と印刷の一致が構造的に保証され、schemaVersionを上げないことで
古いアプリ・古いデータの双方向互換を保つ。

## ADR-023: overlayのストローク入力は `bar.strumOverride` と既存 strumPatterns へ合流する（R2）

- 日付: 2026-07-08 / 状態: 採用

**決定**: R2では `.slot-overlay` の `strum` 行に入力欄を置き、
保存先は新フィールドを増やさず既存の `bar.strumOverride` と `project.strumPatterns` を使う。
入力欄が空なら基本ストローク継承、値があれば小節専用パターンへ正規化して上書きする。

1. **保存構造は据え置き**: 文字列を別フィールドへ保存しない。記号入力はR1と同じ
   パーサーで slots へ正規化し、`schemaVersion` は1のまま維持する。
2. **小節別上書き**: `setBarStrumText()` が `strum_bar_<barNumber>` 系の小節専用IDを

   を `null` に戻し、未参照の小節専用パターンは掃除する。
3. **紙面表示**: A4紙面のストローク段は `bar.strumOverride` を最優先し、
   ない場合だけユーザー入力済みの基本ストロークを継承する。基本ストローク未入力なら

   基本ストロークを示す。上書き中は入力欄に小節別ストロークを表示し、
   空欄へ戻すと基本継承へ戻る。
5. **既存経路維持**: プレDTMの `resolveProjectStrumPattern()` はすでに
   `bar.strumOverride || project.basicStrumPatternId` を読むため、伴奏生成側は無変更でよい。

**理由**:
- ストロークもコード/歌詞/ドレミと同じく「既存モデルへ合流」が最も安全で、
  保存/復元/JSON/プレDTMへの影響を最小にできる
- A4紙面上の直接編集感は足しつつ、紙面DOMを `input` 化せず印刷物として保てる
- 小節専用パターンを `strumPatterns` に閉じることで、将来の記譜化やMIDI化でも
  既存のストロークデータ経路をそのまま使い回せる

## ADR-025: UI再編は「譜面全幅化（D1）→ 下部ドック型スロットエディタ（D2〜D3）」の順で段階導入する。案Aを推奨方針とする

- 日付: 2026-07-10 / 状態: 採用

**背景**: 実機確認で、曲情報フォーム・セクション管理・小節グリッド一覧が常設で左カラムを占有し、
A4紙面プレビューが左右2ペインの残り幅に押し込まれて縮小表示され、編集しづらいことが判明した。
Fable5による設計レビューの結果、案A「下部ドック型スロットエディタ」（紙面下部に拍/裏拍/16分の
入力ドックを常設し、紙面はクリックで選択するだけにする）を将来方針として推奨された。

**決定**: 下部ドック型スロットエディタ本体（案A）へ一気に飛ばず、まず土台としてD1を先行実装する。

1. **D1（完了・v0.16.0。push済み: `19ea503e`）**: レイアウト再編のみ。
   `.sheet-layout` を左右2ペインから1カラムへ変更し、A4紙面プレビューを画面幅いっぱいに近い
   状態で主役表示する。曲情報・セクション・表示設定・小節グリッド（旧一覧編集）は、
   上部の折りたたみドロワー（`<details>`）へ移し、既定で閉じておく。
   小節グリッド（旧一覧編集カード）はドロワー内でさらにネストした`<details>`に収め、
   常設表示から外す（削除はしない。データ・機能は維持）。
   紙面上の小節クリック選択・ミニoverlay編集（S1b〜R2で実装したコード/ストローク/歌詞/ドレミ
   入力、Alt+←/→、まとめて入力）は暫定維持し、ロジックは変更しない。
   データ構造・保存/復元/JSON/プレDTM/MIDI経路は無変更、`schemaVersion` は1のまま。
2. **D2 MVP（本決定でスコープ確定・v0.17.0で実装）**: 下部ドックの「器」だけを先に作る。
   小節上に出ていた極小の `.slot-overlay`（S1b〜R2）を画面下部固定の横長ドックへ移設し、
   入力欄を明らかに大きく・見やすくする。**拍/裏拍/16分セル入力（本格スロット入力）は
   D3へ送り、D2では現行の1小節1入力欄（コード/ストローク/歌詞/ドレミの4行＋まとめて入力）を
   そのまま大きな器に収めるだけに留める。** `_proto/slot-editor.*` のスロット入力UIは
   移植しない（参考にも今回はしない。D3で改めて参照する）。
   実装は「完全置換ではなく最小変更で移行」の方針を取り、`#sc-slot-overlay` のDOM位置を
   紙面プレビュー内から画面下部固定表示へ変更し、内部の行構造・入力ロジック
   （`applyOverlayFieldChange` 等）はS1b〜R2からそのまま流用した。
   選択ハイライト・印刷除外（`.slot-overlay { display:none!important }` in print.css）は維持。
   データ構造・保存/復元/JSON/プレDTM/MIDI経路は無変更、`schemaVersion` は1のまま。
3. **D3（未着手）**: ドックの拍/裏拍/16分セル入力への拡張。ここで初めて
   `_proto/slot-editor.*` のスロット入力UIを参考にする。
4. **VexFlow / 五線譜 / MusicXML（Phase 6）は、D1〜D3のUI再編が完了してから着手する。**
   紙面レイアウトの主従関係が定まる前に記譜エンジンを入れると、座標系・DOM構造の
   手戻りコストが大きいため。

**理由**:
- 下部ドック本体は「拍/裏拍/16分セル入力」という新しい入力パラダイムであり、
  紙面のレイアウト（主役化）となるべく同時に変えると、検証すべき変数が増えて
  問題の切り分けが難しくなる。まずレイアウトだけを独立して安全に検証する
- D1は既存のoverlay編集・小節グリッド一覧・保存/JSON/プレDTM/MIDIのロジックを
  一切変更せず、DOMの配置（どこに何があるか）だけを変えるため、リグレッションの
  リスクを最小に保ったまま「譜面が主役」という次フェーズの前提を先に作れる
- 小節グリッド一覧を削除せず折りたたみに留めることで、下部ドックが未完成の間も
  既存の編集手段（一覧UI・overlay）を失わない
- D2でも「器の拡大」と「入力パラダイムの刷新（スロット入力）」を同時にやらないことで、
  D1と同じ理由（検証変数を絞る）によりリグレッションリスクを最小化できる。
  D2の時点で既存入力ロジックを丸ごと流用できたため、実装量・リスクとも小さく抑えられた

## ADR-026: 選択小節拡大エディタは「選択小節のX線写真」＝タイムグリッド・ドックとして実装する。D3aはストロークのみセル編集する

- 日付: 2026-07-10 / 状態: 採用

**背景**: D2実機確認で、下部ドックが画面下半分近くを占有するただの大きな縦積みフォームに

- まとめて入力が常時右カラムに表示され、通常の入力を邪魔する
- 「選択小節を拡大した感覚」がなく、紙面の小節との対応関係が薄い

Fable5による設計レビューの結果、「選択小節のX線写真」方針が確定した。紙面上の選択小節と
同じ段順・同じ時間軸を保ったまま横方向へ拡大し、各位置を編集できるタイムグリッド・ドックにする。

**決定**: D2の巨大フォーム型ドックは不採用とし、タイムグリッド・ドックへ置き換える。
ただし本格的な「拍位置ごとのセル編集」は、まずストローク行だけに限定してD3aで実装し、
歌詞・ドレミのセル編集はD3bへ送る。

1. **ドックの器（D3a・v0.18.0で実装）**:
   - `position: fixed; bottom: 0;` の器はD2から維持しつつ、高さを画面下半分近く
     （D2は `max-height: 52vh`）から **約200〜220px（`max-height: 240px`）** へ大幅に
     コンパクト化する。`body.dock-open` の予備余白も `56vh` → `260px` に縮める
   - ヘッダーは1行に集約する（小節番号・セクション名・前へ/次へ・8分/16分・
     基本に戻す・まとめて入力・Alt+←/→ヒント・閉じる）。狭い画面で折り返して
     ドックが急激に高くなることを避けるため、ヘッダーは横スクロール可能にする
   - コード/ストローク/歌詞/ドレミを、紙面と同じ段順のまま共通の拍ルーラーの下に並べる。
     境界線の強弱（小節端＞拍頭＞8分位置＞16分位置）は、紙面のタイミンググリッド
     （`sheet-renderer.js` の `buildTimingGrid`）と同じ判定基準をドック側にも複製する
     （ロジックは小さいpure関数のため複製、共有関数化はしない）
2. **セル編集の対象はストロークのみ（D3a）**:
   - コードは従来どおり1小節1コードの大きな1セル（`setBarChord()` を継続使用）。
     DOM構造は将来の拍単位入力に拡張できる形にしておくが、今回は実装しない
   - 歌詞・ドレミは見た目だけ時間軸上のグリッドに揃えるが、入力方式は
     現行の小節単位文字列入力のまま（`setBarLyricsText()` / `setBarDoremiText()`）。
     8個のセルのように見せすぎない（誤解を防ぐため、枠を分割しない1つの入力欄にする）
   - ストロークだけは実際に拍位置ごとのセル編集にする。`song-model.js` に
     `setBarStrumSlots(project, barNumber, slots)` を新設し、小節専用パターンIDの
     allocate・衝突回避・未参照掃除といった既存ロジックをここに共通化する。
     `setBarStrumText()` は「文字列パース→slots生成→`setBarStrumSlots()`」という
     薄いラッパーへ変える（保存構造・パターンID管理ロジック・`schemaVersion` は無変更）
   - セルの解像度（8分/16分）は「ストロークslotsが必要とする解像度」と
     「ユーザーが一時的に選んだ解像度」の大きい方を使う。手動選択は
     barNumberをキーとするUI一時状態（`_proto/slot-editor.js` の `barResOverride` と
     同じ「小節ごとに憶えておく」思想）とし、保存しない・schemaVersionを変えない
3. **まとめて入力はポップオーバー化（D3a）**: 常設右カラムを廃止し、ヘッダーの
   ボタンから開くフロート型ポップオーバーにする。中身（コード/歌詞/ドレミのまとめて
   入力ロジック）はD2までのものをそのまま再利用し、ストロークのまとめて入力は追加しない
4. **選択小節の自動スクロール（D3a）**: 小節クリック・前へ/次へ・Alt+←/→の直後、
   選択小節がドックの裏に隠れる場合だけ自動スクロールする。入力中の再描画毎には行わない
5. **D3b（未着手）**: 歌詞・ドレミのtick位置セル編集。旧・小節グリッドカードとの
   データ競合（旧UIで編集するとtick:0の単一イベントへ潰れる）への対応が必要
6. **D3c（未着手）**: まとめて入力の自動分解配置、旧・小節グリッドカードの撤去、
   UI全体の整合仕上げ

**理由**:
- D2で判明した「ただの大きなフォーム」問題の根本原因は、時間軸という譜面編集の
  本質的な構造をUIが表現していなかったことにある。タイムグリッド化はこの構造を
  UIに反映させる直接的な解決策であり、Fable5レビューの「選択小節のX線写真」という
  比喩がその設計意図を最も端的に表す
- 歌詞・ドレミも一緒にセル編集化すると、tick位置への分解ルール（モーラ分解・音節結合等）
  の設計判断が同時に必要になり検証範囲が急拡大する。ストロークは既にD1〜D2で
  `bar.strumOverride` 経由の小節別上書きが確立しており、セル編集への拡張が
  データ構造・保存経路の両面で最も安全に着手できる対象だった
- `setBarStrumSlots()` への共通化は、将来のD3b拡張（歌詞・ドレミ）やD3c（bulk自動分解）が
  同じ低レベルAPIパターンに乗れる土台にもなる
- 「器のコンパクト化」と「ストロークのセル編集化」は同じD3aで同時に行ったが、
  これはD1/D2のような分割とは異なる判断である。両者は「選択小節のX線写真」という
  単一のUI言語（時間軸グリッド）を実現するために不可分であり、器だけを先に
  コンパクト化しても中身が旧フォームのままでは意味がないため、まとめて実装した

## ADR-027: 選択小節エディタを紙面デザインのフローティング小節ルーペとする（F1）

- 日付: 2026-07-10 / 状態: 採用

**背景**: D3a実機確認で、タイムグリッド編集そのもの（拍ルーラー・8分/16分・ストローク
セル編集）は機能的に良かったが、「黒い編集パネル・画面下固定・紙面と色/罫線/フォントが
異なる」ため、「選択した小節そのものを拡大した表示」には見えないという課題が判明した。

**決定**: D3aのタイムグリッド編集機能はそのまま流用し、表示方式・デザイン・位置・サイズ
だけをF1として変更する。小節ルーペを「白い紙面の上に浮かぶ、もう一枚の小さな紙。選択小節
を切り出して拡大したように見える」ものと定義する。

1. **D3aの機能を維持**: 拍ルーラー、8分/16分切替、コード（1小節1セル）/ストローク
   （本格セル編集）/歌詞/ドレミの時間軸配置、基本ストローク継承/override/基本に戻す、
   `setBarStrumSlots()`、まとめて入力、前へ/次へ、Alt+←/→、showStrum、グリッドON/OFF
   は一切変更しない。歌詞・ドレミのセル編集（F2/F3）・旧小節グリッドカード撤去（F4）は
   今回含めない。
2. **fixed bottomは不採用。position: fixedの自由配置へ**: `bottom: 0` /
   `transform: translateX(-50%)` を廃止し、`left` / `top` のpx指定で自由配置する。
   初期位置は水平中央・垂直はビューポート52%（`viewportHeight * 0.52`）。
   `body.dock-open` の `padding-bottom`（紙面下部の予備余白確保）はフローティング化に
   より不要になったため撤去した。
3. **紙面と同じアイボリー・罫線・文字色**: `.sheet-page` 系の実値（背景 `#fdfbf5`、
   墨色 `#2a2620`、コード赤 `#c0392b`、ドレミ青 `#4a6fa5`、ストローク茶 `#754a14`、
   タイミンググリッドの線色等）を `theme.css` の `:root` に `--paper-*` トークンとして
   切り出し、`.sheet-page` 側もこのトークンを参照するようリファクタリングした
   （値は完全に同一のため、画面・印刷の紙面の見た目は変化しない）。ルーペ側の
   コード/歌詞/ドレミ/ストローク/拍グリッドは、これらのトークンを直接使う。
   外枠だけは選択小節と同じアンバー2px実線とする。
4. **タイトルバードラッグ**: 既存のヘッダー（`.slot-overlay-head`）自体をドラッグ領域にし、
   Pointer Eventsで実装する。button/input/select/textarea/a・ポップオーバー・
   リサイズハンドルの上ではドラッグを開始しない。
5. **右下ハンドルでの比例リサイズ**: `resize: both` は使わず、Pointer Eventsで幅だけを
   変更する。`--bar-loupe-scale`（=現在幅/960）をCSS変数として更新し、フォントサイズ・
   行高さ・セルサイズ等を比例させる。ストロークセルのクリック領域・フォントサイズには
   `max()` で最小フロアを設け、16分表示でも操作不能にならないようにする。高さは
   内容に応じた自動のままとし、単独では変更できない仕様にする。
6. **appSettingsへ位置/幅を保存**: `cruiseStudio.appSettings` の `barLoupe` キー
   （`{x, y, width}`）へ、ドラッグ終了時・リサイズ終了時にのみ保存する。既存の
   `storage.getAppSetting`/`setAppSetting` をそのまま使う。projectデータには入れず、
   `schemaVersion` も変更しない。開閉状態・選択中小節・8分/16分の一時表示・
   まとめて入力ポップオーバーの開閉は保存しない。
7. **小節移動時は位置・サイズを維持**: 前へ/次へ・Alt+←/→・紙面上の別小節クリックの
   いずれでもルーペのx/y/width/scaleは変わらない。`renderSlotOverlayContent()` が
   `innerHTML=''` で子要素だけを作り直す既存構造を活かし、ルート要素自身の位置は
   毎回 `applyLoupeGeometry()` で再適用するだけにとどめ、初期化ロジックを再実行しない。
8. **狭い画面（680px未満）はPCファーストの範囲でフォールバック**: 幅を強制し、
   リサイズハンドルを隠す程度にとどめ、大規模なモバイル最適化は行わない。

**理由**:
- D3aで確立したタイムグリッド編集ロジック（データ構造への読み書き・キーボード操作・
  解像度判定）は正しく機能しており、崩す理由がない。今回変えるべきは「見た目と置き場所」
  だけであり、スコープを表示・操作系に限定することでリグレッションリスクを最小化できる
- 「紙面の上に浮かぶもう一枚の紙」という視覚メタファーは、ADR-011（アプリ全体は黒基調、
  紙面だけは白）の思想をそのまま踏襲しており、既存デザイン言語と矛盾しない
- 色をトークン化して共有することで、将来ルーペのデザインを調整する際に紙面本体を
  誤って変更してしまうリスクを構造的に下げられる
- 位置・サイズをprojectデータではなくappSettingsに保存することで、DATA_MODEL.md /
  ADR-025〜026で確立した「UI状態と曲データの分離」を維持できる

## ADR-028: 歌詞はbar.lyricsのtickイベントを正とし、セル単位で編集する（F2a）

- 日付: 2026-07-10 / 状態: 採用

**背景**: D3a/F1のルーペは、歌詞をコードと同様「小節1つに1つの文字列入力」のまま扱って
おり、`setBarLyricsText()` が常に `tick:0` の単一イベントへ潰して保存していた
（ADR-009時点の暫定仕様）。これでは紙面上でtick位置に応じた歌詞配置（1音節=1マス）が
できず、旧・小節グリッドカードの歌詞inputも同じ関数を呼ぶため、tick位置セル編集を
導入しても旧UIからの編集で簡単に情報が潰れてしまう（`ai-handoff/PROJECT_STATE.md`
9章で既知課題として記録済み）。Fable5による設計レビューで、F2の範囲を「単発セル編集
のみ」に絞ったF2aと、「貼り付け自動展開・自動分割・小節またぎ移動・新まとめて配置方式」
を扱うF2bに分割する方針が確定した。

**決定**: `bar.lyrics` は引き続き `[{tick, text}]` のフラットな配列とし、1セル=1
イベントとして複数保持できるようにする。`schemaVersion` は変更しない
（DATA_MODEL.md 5章のADR-009時点の記述どおり「将来の音節単位入力は同じイベント形の
まま複数イベントへ分割するだけでよい（migration不要）」が実際にそのまま成立した）。

1. **`song-model.js` に新設**:
   - `setBarLyricSlot(project, barNumber, slotTick, slotTicks, text)`: 指定tick範囲
     （1セル分）だけを書き換える。`slotTick` は整数・0以上・小節長未満、`slotTicks` は
     正数で `slotTick+slotTicks` が小節長以下でなければ書き込まず `{ok:false, warnings}`
     を返す。`text` は前後の半角/全角空白をtrimし、空文字・空白のみ・「・」はすべて
     削除入力として扱う。「ー」「〜」は通常の文字として保持し `durationTicks` は
     持たせない（歌詞に音価の概念は導入しない）。model層は `renderPreview` /
     `markDirty` / 保存を一切行わない（UI層の責務として明確に分離する）
   - `getBarLyricSlots(project, barNumber, resolution)`: `bar.lyrics` を指定解像度
     （8 or 16）のセル配列 `{text, eventCount, isMulti}` として読み出す純粋な
     読み取り関数。同一セルに複数イベントがある場合はtick順に連結する
   - `barNeedsSixteenthResolution({strumSlots, lyrics, chords, melody})`: 小節が16分
     表示を必要とするかの共通判定関数。tick%240!==0（またはdurationTicks%240!==0）の
     イベントが1つでもあれば16分を要求する。`sheet-renderer.js` の
     `barNeedsSixteenth(bar, strumSlots)` は元の判定ロジック（lyrics/chords/melody/
     strumSlotsすべて考慮）と完全に同じ挙動になるよう、この共通関数への委譲へ置き換えた。
     F3で `melody` 側のtick判定もこの1関数に統合できる設計にしてある
   - `setBarLyricsText()` は削除しない（旧JSON・他呼び出し元との互換のため）。ただし
     ルーペ・旧小節グリッド・まとめて入力からの新規呼び出し経路はすべて断つ
2. **既存tick:0全文データは自動分割しない**: `[{tick:0, text:"..."}]` という
   単一イベントのデータは、セル編集導入後も先頭セルにその全文がそのまま表示され
   （長ければellipsis・title属性で全文確認可）、他セルは空のまま。先頭セル自体は
   通常どおり編集可能。ユーザーが明示的に操作しない限り、勝手な自動分割・自動保存し
   直しは一切行わない（Undoが無いエディタであることを踏まえた安全設計）
3. **歌詞セルUI**: ルーペの歌詞行を、ストローク行と同じ共有時間軸のセル
   （8 or 16マス、`<input>`）へ置き換える。単一文字・複数文字・貼り付けは、すべて
   そのセルの `text` へそのまま保存する（F2aでは自動分割・自動展開はしない＝
   1操作は必ず「今のセルの範囲」だけを変更する、というUndoレス安全設計の原則を
   歌詞編集にもそのまま適用する）
4. **IME 3層ガード**: (1) keydownで `event.isComposing===true` または
   `event.keyCode===229` またはそのセルが変換中フラグ中なら、セル移動/確定処理を
   スキップする。(2) `compositionstart` でセルごとの変換中状態を持ち、変換中は
   `markDirty` / `renderPreview` を呼ばない・ルーペを再描画しない（IMEのDOMを壊さない
   ため）。(3) `compositionend` 時に `justComposed` フラグを立て `setTimeout(...,0)` で
   解除し、Safari等でIME確定のEnterが `isComposing=false` として届く場合でも、
   その1回はセル移動用Enterとして扱わない（保存自体は `compositionend` または
   `blur` で行う）
5. **旧・小節グリッドカードの歌詞inputはreadonly化**: `disabled` ではなく
   `readOnly` にする（値は表示・フォーカス可能なまま）。フォーカスすると該当小節を
   選択してルーペを開く。`setBarLyricsText()` への書き込み経路をここから完全に断つ。
   コード・ドレミのinputは引き続きフル機能のまま残す
6. **まとめて入力の歌詞は一時停止**: F2a〜F4の移行期間として、「歌詞」選択時は
   textarea・適用ボタンをdisabled化し、「歌詞はセル編集に移行しました。まとめて配置は
   今後の対応予定です。」と案内する。コード・ドレミのまとめて入力は変更しない
7. **紙面のtick配置表示**: 互換仕様として、`bar.lyrics` が `tick:0` の単一イベント
   のみ（または空）なら従来どおり全文を1つのspanで表示する。それ以外は、小節の
   必要解像度（ルーペと同じ `barNeedsSixteenthResolution` 判定）に応じた8/16分割の
   グリッドへ、各イベントをtick位置ごとに配置する。長い文字列は隣の空セルへ自然に
   はみ出してよいものとし、極端な重なりは許容する（本格的な組版調整はVexFlow等の
   将来フェーズで扱う）
8. **貼り付け自動展開・自動分割・小節またぎ移動・新まとめて配置方式はF2bへ**:
   「1音節ずつ入力する」体験の完成度を上げる機能は、まず単発セル編集の土台が
   安定してから積み上げる

**理由**:
- ストローク（ADR-026/D3a）で確立した「共有時間軸のセルグリッド + tick範囲を
  書き換える薄いsetter」というパターンを歌詞にもそのまま適用でき、検証範囲を
  「1セルの読み書きが正しいか」に閉じ込められる
- `schemaVersion` を上げずに済むのは、ADR-009の時点で「`bar.lyrics` は複数イベントへ
  分割するだけでよい」設計を先取りしていたため。今回はその設計判断が正しかったことの
  実証でもある
- 貼り付け自動展開や自動分割は、日本語の音節分割ルール（モーラ・拗音・促音の扱い等）
  という別の難しい設計判断を含むため、単発セル編集（F2a）と分離することで
  レビュー・検証の粒度を保った
- 旧・小節グリッドカードをreadonly化（撤去ではなく）したのは、値の確認・該当小節への
  ジャンプという価値は残しつつ、tick位置情報を破壊する書き込み経路だけを確実に断つ
  ためである。ドレミ側は同じ問題が未解決のまま残るため、F3で同様の対応が必要になる
  （`ai-handoff/PROJECT_STATE.md` 9章に記録）

## ADR-029: 小節ルーペを拍グループ入れ子DOMへ構造化する（G1）

- 日付: 2026-07-10 / 状態: 採用

**背景**: F2aまでの小節ルーペは、ストローク・歌詞がセル構造（フラットな
`grid-template-columns: repeat(res, 1fr)`）である一方、コード・ドレミは小節全体を
横断する1入力のままだった。将来G2（歌詞の行別/拍別8分/16分切替）・G3（ストロークの
同機能）を安全に実装するには、まず「拍」という単位をDOM構造として明示的に持たせておく
必要がある。今回はその土台整備だけを、機能等価なリファクタとして先に行う。

**決定**: 各行のタイムライン部分を「拍グループ（拍の数だけ均等幅・flex） × 拍内スロット
（解像度なりのスロット数・grid）」の二段構造へ揃える。

1. **共通ヘルパー `populateBeatGroups(container, res, buildSlot)` を新設**
   （`sheet-cruise.js`）。`getBeats(project)` 拍分の `.beat-group`（`flex:1 1 0`）を作り、
   各グループ内へ `slotsPerBeat`（=res/beats）個のスロットを敷き詰める。
   拍ルーラー・ストロークセル・歌詞セル・コード/ドレミ背面線のすべてがこの1関数だけを
   経由することで、拍・8分・16分の境界線のx座標が行をまたいで完全に一致する
2. **セルのDOM順序・data-cell-indexは既存のフラットindexと完全に同じ**にする
   （拍0のスロット0..slotsPerBeat-1 → 拍1のスロット0.. という順で追加するため、
   `querySelectorAll('.dock-strum-cell')` 等の取得順・`focusStrumCellByIndex` /
   `focusLyricCellByIndex` の挙動・IME 3層ガード・フォーカス復元は無変更のまま動く）
3. **コード・ドレミは1小節1入力を維持**。`setBarChord()` / `setBarDoremiText()` の
   経路もセル分割もしない。代わりに `buildTimelineBackground(res)` が、同じ
   `populateBeatGroups()` を使って装飾専用の縦線レイヤーを入力欄の背面へ敷く
   （`pointer-events: none` ・入力欄側は `z-index:1` で前面に出し、クリック・
   キャレット操作を一切妨げない）
4. **境界線の強弱判定は既存の `timelineBoundaryClass()` をそのまま使う**（挙動不変）。
   拍境界 ＞ 8分位置 ＞ 16分位置の順に濃さを付け、紙面と同じ `--paper-line-*` トークンを
   共有する
5. **解像度ロジックは今回変更しない**。ヘッダーの全体8分/16分切替（`getStrumGridResolution`
   / `isBarDataForcedSixteenth`）をそのまま使い、全拍が同じ解像度で描画される
   機能等価リファクタにとどめる。行別・拍別のローカル解像度・8/16ミニボタン・
   `state.loupeBeatRes` 等の状態・`beatNeedsSixteenthResolution()` はG2以降で追加する
6. **拍グループ間の幅ズレ対策**: `.slot-overlay-ruler-track` は他の行（`.dock-strum-grid`
   等）が持つ1px枠（`border:1px solid`）と同じボックスモデルを再現するため、
   透明な `border:1px solid transparent` を追加した。拍ルーラーだけ枠を持たないままだと、
   `.beat-group` へ配分される実効幅が他行より2px広くなり、拍・8分・16分の線が
   サブピクセル単位でズレるため

**理由**:
- G2/G3で「拍単位の解像度切替」を実装する際、DOM構造がすでに拍単位で分かれていれば、
  各 `.beat-group` に解像度状態を持たせるだけで済み、DOM再構築の範囲を局所化できる。
  逆に先にDOM構造を変えずにG2/G3を実装すると、セル再配置とローカル解像度ロジックを
  同時に扱うことになり検証範囲が急拡大する
- `populateBeatGroups()` を全行の共通経路にすることで、「拍・8分・16分の境界線が
  一直線に並ぶ」という見た目の要件を、CSSの微調整ではなく構造的に保証できる
  （同じ関数・同じCSSクラスを使う限り、行ごとに線がズレる余地がない）
- コード・ドレミへ背面線だけを足す設計にしたのは、拍単位コード編集・ドレミのセル化
  （F3）は別フェーズの判断であり、G1では「時間軸を揃えて見せる」以上のことをしない
  という明確なスコープの線引きのため
- 全拍を同一解像度で描画する制約は、G2/G3で `.beat-group` 単位のローカル解像度を
  導入する際の比較対象（回帰確認の基準線）としてそのまま使える

**G1実機確認後の表示修正（同日）**:
1. **歌詞行の縦線が表示されない不具合を修正**: 境界線の強弱色を定義する
   CSSセレクタ（`.is-beat` / `.is-8th` / `.is-16th` / `.is-barend`）が
   `.slot-overlay-ruler-cell` と `.dock-strum-cell` のみを対象にしており、
   `.dock-lyrics-cell` が含まれていなかったため、歌詞セルの `border-right-color` が
   常に基本値の `transparent` のままだった（F2a導入時からの既存不具合。G1で
   縦線を強調する構成に変えたことで見た目上の欠落が顕在化した）。
   `.dock-lyrics-cell.is-beat/.is-8th/.is-16th/.is-barend` を同じセレクタ列へ追加し、
   strumセルと同じ色（`is-barend`も含む）を適用して解決した
2. **拍ルーラーの表示文言を「1/2/3/4」から「1拍/2拍/3拍/4拍」へ変更**:
   `buildBeatRuler()` 内のラベル生成のみを変更。tick計算・`getBeats()` の使用方法・
   線の位置は無変更

**今回明記しておく将来要望（未実装・未確定設計）**: ユーザーから、拍単位コード編集・
ドレミのセル編集・行別/拍別16分→8分切替時のデータ移動・ストローク「〜」候補追加・

## ADR-030: 歌詞行の拍別解像度はWeakMapの一時UI状態とし、ローカル指定を全体設定より優先する（G2b）

- 日付: 2026-07-11 / 状態: 採用

**背景**: G1で拍グループDOM・縦線基盤を整備し、G2aで歌詞セルの位置管理をtick中心へ
移した。G2bでは、小節ルーペの歌詞行で拍ごとに独立して8分/16分表示を切り替えられる
ようにする（例: 1拍目8分・2拍目16分・3拍目8分・4拍目16分のmixed resolution）。

**決定**:

1. **歌詞の拍別ローカル解像度は`bar`オブジェクトをキーにしたWeakMap
   （`state.lyricBeatRes: WeakMap<bar, Map<beatIndex, 8|16>>`）で保持する一時UI状態**とし、
   `project` / `localStorage` / `cruiseStudio.appSettings` のいずれにも保存しない。
   同一セッション内で小節を移動して戻ってきても（`bar`オブジェクトの参照が同じ限り）
   維持されるが、JSON読込・新規project作成・保存後の再読込では`project.bars`が
   新しいオブジェクト配列になるため、古いWeakMapエントリは自然に参照されなくなり
   （GC可能）、ローカル指定は消える。`schemaVersion`は変更しない
2. **実効解像度の優先順位は2段階だけ**: (1) その拍のローカル指定
   （`getLyricBeatLocalRes`） (2) ヘッダーの全体8分/16分設定
   （`getStrumGridResolution`。G2bでストローク由来のみに変更。後述4）。
   ローカル指定が無い拍（＝「自動」）は、常にこの全体設定だけに追従する
3. **歌詞データ由来のauto-16は採用しない**: F2a〜G1では、歌詞にtick%240!==0の
   イベントがあると全体解像度が自動的に16分へ強制されていたが、G2bではこれを廃止した。
   「自動」拍は全体設定にのみ追従し、その拍にたまたま16分位置のデータがあっても
   自動では16分へ昇格しない。16分位置のイベントは、実効解像度が8分であれば
   対応する8分セルへtick順に連結して集約表示される（`getBarLyricSlotRanges`。G2a）
4. **全体8分ボタンのdisabledロックはストローク由来のみに限定**（`isStrumDataForcedSixteenth`。
   旧`isBarDataForcedSixteenth`から歌詞判定を除いたもの）。ストロークに16分位置データが
   ある場合は従来どおり全体8分ボタンをdisabledにし、理由も表示する
   （`getForcedSixteenthReason`）。ストローク行自体の保存・表示・安全仕様は無変更
5. **表示切替（解像度変更）だけではbar.lyricsを一切変更しない**。
   `setBarLyricSlot`はセルが実際に編集・確定されたときだけ呼ばれる
6. **8分集約セルは、編集されて確定された時だけ、その8分範囲のイベントを1つへ統合する**。
   クリックしただけ・フォーカスしただけ・何も変えずblur・Enter/Tabで移動しただけでは
   統合しない。これはG2aで導入した「無変更commitガード」（`input._initialValue`との
   比較）をそのまま流用することで実現している。新たな無変更判定ロジックは追加していない
7. **拍別ローカル解像度ボタン・全体8分/16分ボタンの両方に、解像度変更前の共通安全処理
   （`prepareLyricResolutionChange`）を適用する**: IME変換中は何もしない→
   アクティブな歌詞セルがあれば変更時のみ1回commit（`skipRender:true`）→
   tick・選択範囲をキャプチャ→解像度状態を変更→1回だけ再描画→G2aのtick中心フォールバック
   （同tick→含む→直前→先頭）でフォーカス復元、の順を厳守する。ボタンは`pointerdown`で
   `preventDefault`し、ボタンへのネイティブフォーカス移動（＝編集中セルの先行blur）を
   止める。先行blurが起きると、その委譲commitが`setTimeout`の次tickへ回り、
   同期的な再描画より後回しになって編集内容が消える危険があるため
8. **拍別ボタンのクリックサイクルは、自動からの最初の1クリックが必ず全体設定と逆の
   解像度へ固定されるようにする**（例: 全体8分・自動→クリック→16分固定→クリック→
   8分固定→クリック→自動）。これにより「押したのに見た目が変わらない」という
   混乱を避ける

**理由**:
- WeakMapを使うことで、「project構造は変更しない」という制約を守りながら、
  「小節を移動しても一時指定が消えない」という実用上必要な永続性（セッション内に限る）を
  両立できる。`bar`オブジェクト参照をキーにするのは、既存の`state.dockManualRes`
  （barNumberキーのプレーンオブジェクト）と違い、`schemaVersion`や`barNumber`の
  変更に引きずられず、かつ明示的なクリーンアップ処理を書かなくても小節削除・
  project差し替えで自然にGCされる点で優れている
- 歌詞データ由来のauto-16を廃止したのは、拍別ローカル解像度という「ユーザーが明示的に
  選ぶ」手段ができたことで、「データを理由にUIが勝手に解像度を変える」という
  F2a〜G1時点の暗黙的な挙動がむしろ驚きの原因になり得ると判断したため。データは
  常に元のtickのまま保持され、表示（8分集約 or 16分展開）だけが選択によって変わる、
  という設計の方が「表示はデータのビュー」という本アプリの一貫した思想
  （`APP_CONCEPT.md`）に合致する
- 無変更commitガードをG2aで先に導入していたことで、G2bの「集約セルの非破壊編集」が
  ほぼ無改造で実現できた。これはG2aをG2bの前提整備として独立させた設計判断
  （ADR未記載だがG2aの背景説明どおり）が正しかったことの実証でもある
- ストローク由来の8分ロックのみ維持したのは、ストロークは行全体が単一パターンの
  slots配列で管理されており（`bar.strumOverride` / `project.strumPatterns`）、
  歌詞のように拍ごとに独立した分解ができる構造をまだ持っていないため。ストロークの
  拍別解像度化はG3で扱う
